# v1.1 Update System Design

## Overview

Add automatic update checking and in-app update installation to Paper Suitecase using the Sparkle framework. Includes branching workflow changes for structured release management.

## Branching & Release Workflow

### Branch structure
- `main` â€” release-only, every commit is a tagged release
- `dev` â€” working branch, all PRs merge here
- Feature branches (`feature/xxx`) branch from `dev`

### Release flow
1. Work accumulates on `dev` via PR merges
2. Bump version in `pubspec.yaml` on `dev`
3. Merge `dev` â†’ `main`
4. Tag `main` with version (e.g. `v1.0.1`)
5. Build macOS app, zip, sign with EdDSA
6. Create GitHub Release with zip attached
7. Update `appcast.xml` in website repo with new entry

### Version numbering
- `v1.0.x` â€” patches (bug fixes, small improvements) leading up to v1.1
- `v1.1.0` â€” the update system itself ships
- `pubspec.yaml` format: `version: X.Y.Z+N` where N increments with each build

## Sparkle Integration Architecture

### How it works
Sparkle is a native macOS framework integrated into the Xcode runner (not the Flutter/Dart layer). It checks an appcast XML file for new versions and handles download, signature verification, app bundle replacement, and relaunch.

### Components

**1. Appcast file (`appcast.xml`)**
- Hosted at `https://initialneil.github.io/papersuitecase/appcast.xml`
- Contains version entries with: download URL (GitHub Releases), file size, EdDSA signature, minimum OS version

**2. EdDSA key pair**
- Generated once with Sparkle's `generate_keys` tool
- Private key stored in macOS Keychain (never committed to git)
- Public key embedded in `macos/Runner/Info.plist`

**3. macOS native integration**
- Sparkle framework added via CocoaPods (`macos/Podfile`)
- Configured in `Info.plist`:
  - `SUFeedURL` â†’ appcast URL
  - `SUPublicEDKey` â†’ EdDSA public key
  - `SUScheduledCheckInterval` â†’ `86400` (24 hours)
  - `SUAutomaticallyUpdate` â†’ `NO` (user controls when to install)

**4. Flutter â†” native bridge (`MethodChannel`)**
- Channel name: `com.papersuitecase/sparkle`
- Dart â†’ native methods:
  - `checkForUpdates()` â€” triggers manual check (for settings button)
  - `getUpdateStatus()` â€” returns `{available: bool, version: String?}`
- Native â†’ Dart callbacks:
  - `onUpdateAvailable(version)` â€” called when Sparkle finds a newer version
- Native side: Swift class `SparklePlugin` registered in `AppDelegate.swift`, wraps `SPUStandardUpdaterController`

**5. Signing releases**
- After building zip: `sign_update PaperSuitecase-macOS-vX.Y.Z.zip`
- Output EdDSA signature added to appcast `<item>` entry

## UI Changes

### Settings icon badge (`tag_sidebar.dart`)
- Wrap existing settings `IconButton` in a `Stack`
- Add positioned 8px teal dot (`var(--teal)` / `Color(0xFF4EB8A1)`) in top-right corner
- Dot visible only when `appState.updateAvailable == true`
- Dot clears when user opens settings (`hasSeenUpdateBadge = true`)

### Settings page â€” About section (`settings_view.dart`)
New section at the bottom of the settings ListView:

- **Current version**: `v1.1.0` (read from package info)
- **Update status line**:
  - "Up to date" (gray) when no update available
  - "Version X.Y.Z available" (teal) when update found
- **"Check for Updates" button**: always visible, calls `checkForUpdates()` via MethodChannel
- **"Install Update" button**: appears only when update available, triggers Sparkle's standard update flow (native download/verify/replace/relaunch dialog)

Uses existing `_SectionHeader` and `_SettingBox` patterns from the current settings view.

### AppState additions (`app_state.dart`)
```

```

### Native side files
- `macos/Runner/SparklePlugin.swift` â€” MethodChannel handler, SPUStandardUpdaterController wrapper
- `macos/Runner/AppDelegate.swift` â€” register SparklePlugin channel

## Update check flow

1. App launches â†’ `AppDelegate` initializes Sparkle's `SPUStandardUpdaterController`
2. Sparkle automatically checks appcast (on launch + every 24h per `SUScheduledCheckInterval`)
3. If newer version found â†’ native sends `onUpdateAvailable(version)` to Dart via MethodChannel
4. `AppState` sets `updateAvailable = true`, `latestVersion = version` â†’ teal dot appears on settings gear
5. User opens settings â†’ `markUpdateBadgeSeen()` clears dot, About section shows version info
6. User taps "Install Update" â†’ calls native `checkForUpdates()` which opens Sparkle's standard update dialog (shows release notes, download progress, install & relaunch)
7. User taps "Check for Updates" â†’ same native call, Sparkle shows "up to date" or update dialog

## Release build pipeline

### One-time setup
1. `generate_keys` â†’ creates EdDSA key pair, stores private key in Keychain
2. Add public key to `macos/Runner/Info.plist` as `SUPublicEDKey`
3. Add `pod 'Sparkle'` to `macos/Podfile`, run `pod install`
4. Create initial `appcast.xml` in website repo (`initialneil/papersuitecase`)

### Per-release steps (manual)
1. Bump version in `pubspec.yaml` (version + build number)
2. Merge `dev` â†’ `main`, push
3. Tag: `git tag vX.Y.Z && git push --tags`
4. Build: `flutter build macos --release`
5. Zip: `ditto -c -k --keepParent paper_suitecase.app PaperSuitecase-macOS-vX.Y.Z.zip`
6. Sign: `sign_update PaperSuitecase-macOS-vX.Y.Z.zip` â†’ copy signature
7. Update `appcast.xml`: add new `<item>` with version, download URL, signature, file length
8. Create release: `gh release create vX.Y.Z PaperSuitecase-macOS-vX.Y.Z.zip --title "..." --notes "..."`
9. Push appcast update to website repo

Automation via GitHub Actions is out of scope for v1.1 but could be added later.

## Out of scope
- GitHub Actions CI/CD for automated builds
- Windows/Linux support
- Delta updates (Sparkle supports these but full zip replacement is fine to start)
- Apple code signing / notarization (can be added independently)
- Release notes display within the app (Sparkle's native dialog shows these)
